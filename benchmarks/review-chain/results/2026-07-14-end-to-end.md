## The 50s are caused by the fixtures, not by the rule — and this is now measured, not argued

`2026-07-13-angles.md` recorded the soft-delete demotion as an **open question** with two readings:
the rule is correct, or the rule is too blunt and the fixture's missing model file is the real cause.
It said the second reading was plausible and **unverified**. It is now verified.

**Diagnostic (`soft-delete-readable`, one scorer, same claim, same prompt):** the fixture rebuilt so
that `app/models/watchlist.py` — carrying the ordinary soft-delete convention — is present in **both**
the base and head commits. The diff under review is **byte-identical** to the original fixture's
(same blob hashes `a94a655..3095c4e`); the model file is not part of it. One variable changed: the
model is readable.

| Tree | Score | Scorer's own reason |
|---|---|---|
| fixture as shipped (model absent) | **50** | *"the Watchlist model file is not readable from this tree… the bug is unknown, not confirmed"* |
| **same diff, model readable** | **100** | *"omits the soft-delete check explicitly mandated by the model's comment ('MUST be excluded from all user-facing reads')"* |

**Conclusion: the cap fired because the fixture is a single-file tree with no model, not because the
rule is wrong.** In any repository where the model file exists — i.e. every real one — this finding
scores 100 and enters the fix-loop. The pipeline's effective recall on this defect is **3/3**, and
the 2/3 above is a property of the benchmark's context-starvation, not of the skill.

This is a diagnostic, recorded as one. It is **not** the fixture's score, and it does not replace it.
The scored fixture result stands at 50 → advisory.

## What the two advisory routings actually mean

Neither advisory finding is discarded. Both are recorded and surfaced to the human — which is the
whole content of the claim "a score below threshold does not mean the finding is false."

## Caveats & limitations (honesty rules)

1. **One scorer per location, n = 1.** No repeat sampling; a single scorer's judgment is not proof of
   the scorer's reliability, only of its behavior on this input.
2. **FIND is the 2026-07-13 run.** This measures SCORE over that FIND output. It does not re-establish
   FIND's recall, which stands at 3/3 from that file.
3. **Angles `call-site-impact` and `prior-art` remain unmeasured** — unchanged from
   `2026-07-13-angles.md`. The fixtures are single-file trees with no `docs/solutions/`.
4. **One scorer died mid-run on an API stall** (`intent-gap:22`) and was re-dispatched. The failed
   attempt produced no score, so nothing was discarded or re-rolled — this is not a
   re-run-until-it-passes event.
5. **The diagnostic tree is synthetic.** I wrote `app/models/watchlist.py` myself, with a soft-delete
   comment that a real model would plausibly carry. A less explicit model file might score lower. What
   the diagnostic proves is *readability changes the score on this diff*, not that every real repo
   yields 100.
