## Notes

The safety-critical result is clean: **all 3 hard-gate fixtures (auth, high-blast hook, data-loss
migration) were correctly forced to `high-risk` and escalated** — no downgrade. The two misses are
on the *judgment* boundary between adjacent lanes, and both are partly **answer-key doubts**, not
clear skill failures (recorded per the honesty rule — the first run is the record, fixtures are
not re-run to pass):

- **add-validation (tiny vs normal):** the blind run classified it `tiny`, reasoning only flag 8
  fires marginally. But the request adds a **unit test** alongside the service change → **2 files**,
  and the lane rule is `tiny` only when `≤1 file & no new public callable`. So `normal` (truth) is
  defensible and the skill under-counted the file span. *Candidate skill finding:* intake did not
  weigh the "≤1 file" tiny-gate against the added test file. Worth a fixture-owner adjudication.
- **multi-domain (normal vs high-risk):** truth asserted `4+ flags → high-risk`; the blind run
  counted only 2 (existing-behavior + multi-domain) and chose `normal` with `medium` confidence.
  This is a genuine **answer-key doubt** — whether "compute quota across two internal services"
  reaches 4 flags is arguable (data-model? weak-proof?). The fixture's expected label may be too
  aggressive; flag it for revision rather than scoring it purely as a skill miss.

- **Scope reminder:** measures only `/feature-intake` against these 7 fixtures — not other skills,
  not the full chain, not real-world rate (`not_observed != absent`). n=7; treat the numbers as a
  claim about this fixture set only.
