## Claim discipline (scope of this result)

Each row is a claim about **one context, one Claude Code version (2.1.217), one HEAD sha**. It is
not evidence about other versions, other rules, or contexts not probed here. `not_observed !=
absent`. The three child-context rows are the **load-bearing** results (they are where the #141
P1 escape lives); the main-session row is contaminated this run (see its note) and is recorded
`unconfirmed`, which — because main is NOT a load-bearing delivery boundary for isolated dispatch
— is a methodological limitation to re-run in a fresh session, not a shipment blocker.

## Headline finding

All three isolated child contexts (implementer, reviewer, scorer) behave **exactly as the P1
lesson predicts**: a path-scoped rule is delivered when — and only when — the dispatch prompt
explicitly Reads it, and a rule that is merely *referenced* (never Read, no `specs/**` file read)
is genuinely **absent** from the fresh context (`NOT IN CONTEXT`). This empirically confirms that
the #141 escape was real: **the escape-relevant direction is the negative control** — a
referenced-but-unread path-scoped rule — and it is cleanly `not-delivered` in **all three** child
contexts. The positive direction (an explicit `FIRST: Read` delivers the rule) is confirmed
`delivered` for the two capable-model contexts (implementer, reviewer, both demonstrably Read the
file); for the cheap-model scorer the positive is recorded **`unconfirmed`** — its heading is
guessable from the filename, so per the README cheap-model caution a Read cannot be proven from the
output. This unconfirmed is on the *positive* (does an explicit Read work on haiku), NOT on the
escape-relevant negative, so it does not reopen the P1 class; it flags that a definitive
cheap-model positive needs a non-guessable marker (see Follow-up).

**Main-session contamination note.** This baseline was produced by the orchestrator that had, in
the same session, executed the subagent-driven-development Step-0, which **explicitly Reads**
`rules/auto-correct-scope.md` AND `rules/wave-parallelism.md`. It had also read multiple
`specs/**/PLAN.md` files (which path-inject `wave-parallelism.md`, `paths: ["specs/**/PLAN.md"]`).
So both rules were already in context via legitimate reads, and the negative control cannot be
honestly answered `NOT IN CONTEXT`. Per the README claim-discipline, this is recorded
`unconfirmed`, **not** silently mitigated. A clean main-session probe requires a **fresh session**
that reads only a `specs/**` non-`PLAN.md` file (e.g. `SUMMARY.md`) and neither rule directly —
re-run there to confirm the path-scoped injection fires and is genuinely scoped. This does not
affect the child-context conclusions, which are the load-bearing ones and were run in fresh
contexts.

## Follow-up (non-blocking)

- **Cheap-model positive marker.** The scorer positive is `unconfirmed` only because
  `# Auto-Correction Scope` is inferable from the filename. A future probe revision should use a
  positive marker that cannot be guessed — e.g. "quote the first sentence under the Rule 3
  heading" — so a cheap-model Read is provable. Tracked here, not blocking: the escape-relevant
  negative-control verdict is clean (`not-delivered`) for all three child contexts.

## Re-probe triggers (staleness)

Re-run this protocol and write a new dated results file when any holds: Claude Code is upgraded
past 2.1.217; the dispatch/subagent context mechanism changes; or a rule's `paths:` frontmatter or
the Step-0 Read wiring changes. A boundary that can no longer be reproduced downgrades to
`unconfirmed` and blocks per the README.
